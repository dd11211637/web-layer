# 协作规范

本文档约定 Web 层 Q1 阶段的分支、提交、联调和契约维护方式。

## 1. 分支约定

- `main`：稳定主分支，用于演示和最终提交；
- `feature/chat-ui`：页面布局、输入交互、消息组件；
- `feature/api-stream`：接口调用、流式输出、引用渲染；
- `docs/module1-refine`：模块文档、测试记录、验收清单；
- `fix/*`：问题修复分支。

## 2. 提交规范

推荐提交信息格式：

```text
feat: add chat input
feat: implement stream response reader
fix: handle empty query
docs: update task checklist
test: add manual q1 cases
```

常用类型：

- `feat`：新增功能；
- `fix`：修复问题；
- `docs`：文档修改；
- `style`：样式调整；
- `refactor`：代码重构；
- `test`：测试相关；
- `chore`：目录、配置或非业务修改。

## 3. 日常开发流程

开始开发前：

```bash
git checkout main
git pull
git checkout -b feature/your-task
```

提交修改：

```bash
git status
git add .
git commit -m "feat: add chat layout"
git push -u origin feature/your-task
```

## 4. 模块边界

Web 层负责：

- 聊天页面；
- 输入校验；
- Agent 接口调用；
- 流式响应解析；
- 答案、引用和状态展示；
- Mock 数据和 Demo 兜底。

Web 层不负责：

- 文档解析；
- 检索召回；
- Prompt 编排；
- LLM 调用；
- 向量库写入或查询策略。

## 5. Contract Owner 规则

模块负责人作为 Web-Agent Contract Owner，统一维护：

- `docs/接口约定.md`；
- `docs/Mock数据说明.md`；
- `src/types/chat.ts`；
- `mock/*.json`；
- 联调测试记录。

任何字段变化必须同时更新文档、类型、Mock 和测试记录。

## 6. 合并前检查

- 页面可以正常打开；
- 输入框可以发送问题；
- Mock 或后端接口能返回回答；
- 流式输出或模拟流式输出正常；
- 引用来源能展示；
- 空输入、无相关文档、检索异常、模型异常和网络异常都有提示；
- `trace_id` 能记录或展示；
- 没有明显控制台报错。

## 7. 冲突处理原则

- 不直接覆盖他人代码；
- 先确认冲突文件中双方修改目的；
- 保留双方需要的逻辑，再本地运行验证；
- 冲突较大时先沟通接口字段和组件边界；
- `docs/接口约定.md` 和 `src/types/chat.ts` 以 Contract Owner 最新版本为准。
